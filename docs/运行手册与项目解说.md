# B07 RAG 校园智能问答助手运行手册与项目解说

本文档写给第一次接触 RAG 的同学。你可以先照着运行，再慢慢理解每个文件和每一步的作用。

## 1. 这个项目到底是什么

这个项目做的是一个“校园智能问答助手”。用户输入一个校园问题，比如“学生证丢了怎么补办？”，系统不会直接凭空回答，而是先去校园知识库里查资料，再根据查到的资料生成答案。

RAG 的全称是 Retrieval-Augmented Generation，中文一般叫“检索增强生成”。

可以把它理解成三步：

1. 用户提问。
2. 系统先从知识库里找最相关的资料。
3. 系统根据这些资料组织答案，并标出引用来源。

它和普通大模型问答的区别是：

普通大模型问答像是“凭记忆回答”，容易编造。  
RAG 问答像是“先查资料再回答”，更适合学校制度、办事流程、图书馆时间、宿舍规定这类需要准确信息的问题。

本项目默认不依赖 API Key，也能本地运行。默认回答方式是“抽取式回答”，也就是从检索到的知识片段里组织答案。以后如果有通义千问或 OpenAI 兼容 API Key，也可以切换到大模型生成回答。

## 8. RAG 的完整流程

本项目的数据流可以理解为：

```text
原始知识库
  -> 数据清洗
  -> 文本切分
  -> 建立检索索引
  -> 用户提问
  -> 检索相关片段
  -> 生成回答
  -> 评测与可视化
```

对应脚本是：

```text
data/raw/campus_faq.jsonl
  -> scripts/01_prepare_data.py
  -> scripts/02_build_index.py
  -> cli.py / app.py / gradio_app.py
  -> scripts/03_evaluate.py
```

### 8.1 知识库

知识库就是系统能查到的资料。  
当前知识库文件是：

```text
data/raw/campus_faq.jsonl
```

每一行是一条 JSON 数据，大概长这样：

```json
{
  "doc_id": "KB001",
  "category": "图书馆",
  "title": "图书馆开放时间",
  "content": "图书馆周一至周五开放时间为...",
  "source": "校园办事指南",
  "url": "https://example.edu/library",
  "last_updated": "2026-05-10"
}
```

其中最重要的是：

- `doc_id`：资料编号，评测时用来判断检索是否命中。
- `category`：类别，比如教务、图书馆、宿舍。
- `title`：标题。
- `content`：正文，真正被检索和回答使用的内容。
- `source`：资料来源，用来做引用。

### 8.2 数据清洗

清洗就是把原始数据整理成更规范的格式，包括：

- 去除多余空格。
- 删除重复条目。
- 检查必填字段是否存在。
- 删除内容太短的无效条目。
- 统计类别数量和文本长度。

对应文件：

```text
campus_rag/data.py
scripts/01_prepare_data.py
```

输出文件：

```text
data/processed/knowledge_base_clean.csv
logs/data_profile.json
```

### 8.3 文本切分

RAG 不会直接把整篇资料全部塞给模型，而是会把资料切成较小的文本块。  
这样检索时可以更精确地找到某一小段相关内容。

对应文件：

```text
campus_rag/splitter.py
```

输出文件：

```text
data/processed/chunks.csv
```

当前 FAQ 内容较短，所以多数资料会对应一个文本块。

### 8.4 检索

检索就是“根据用户问题，在知识库文本块里找最相关的几段”。

本项目实现了三种策略：

| 策略 | 解释 | 优点 | 局限 |
|---|---|---|---|
| TF-IDF | 把文本转成词频权重向量，再算相似度 | 简单、稳定、无需联网 | 对深层语义理解有限 |
| BM25 | 经典关键词检索算法 | 对关键词命中敏感，速度快 | 问法变化大时可能漏掉 |
| Hybrid | TF-IDF 和 BM25 加权融合 | 兼顾相似表达和关键词命中 | 比单策略略复杂 |

对应文件：

```text
campus_rag/retriever.py
scripts/02_build_index.py
```

索引文件：

```text
models/tfidf_vector_store.joblib
```

### 8.5 生成回答

检索到资料后，系统会基于最相关片段生成回答。

默认方式是抽取式回答：

- 找到排名第一的资料。
- 用资料里的内容组织回答。
- 列出参考资料编号。
- 顺便显示其他相关资料。

对应文件：

```text
campus_rag/generator.py
```

如果配置了 API Key，也可以调用大模型，让大模型根据检索资料生成更自然的回答。

### 8.6 无检索基线

无检索基线就是“不查知识库，直接回答”。  
本项目保留它是为了做对比实验。

老师通常很看重对比实验，因为它能说明：

- RAG 是否真的比普通回答更可靠。
- 检索资料是否能提高答案准确率。
- 不同检索策略谁更好。

本项目的无检索基线回答比较泛化，所以关键词召回很低，这能突出 RAG 的优势。

### 8.7 评测

评测问题在：

```text
data/raw/eval_questions.jsonl
```

每个评测问题包含：

- `question`：测试问题。
- `gold_doc_id`：标准答案应该命中的知识库编号。
- `answer_keywords`：回答里应该出现的关键词。

评测脚本：

```text
scripts/03_evaluate.py
```

评测指标：

| 指标 | 含义 |
|---|---|
| Hit@1 | 排名第一的检索结果是否就是标准资料 |
| Hit@3 | 前三个检索结果里是否包含标准资料 |
| MRR | 标准资料排得越靠前，分数越高 |
| keyword recall | 回答覆盖了多少标准关键词 |
| avg latency | 平均检索耗时 |

## 9. 每个文件和文件夹有什么用

### 9.1 顶层文件

| 文件 | 作用 |
|---|---|
| `README.md` | 项目简介、分工、基础运行方式 |

### 9.2 `campus_rag/`

这是项目核心代码目录。

| 文件 | 作用 |
|---|---|
| `config.py` | 统一保存项目路径，比如数据、模型、日志、报告路径 |
| `data.py` | 读取 JSONL、清洗知识库、统计数据概况 |
| `splitter.py` | 把知识库正文切成文本块 |
| `retriever.py` | 实现 TF-IDF、BM25、Hybrid 三种检索方式 |
| `generator.py` | 根据检索结果生成 RAG 答案，也支持调用大模型 |
| `evaluate.py` | 计算 Hit@1、Hit@3、MRR、关键词召回等评测指标 |
| `loaders.py` | 从 PDF、DOCX、TXT、MD 中抽取文本 |
| `__init__.py` | 让 `campus_rag` 成为可导入的 Python 包 |

### 9.3 `scripts/`

这是项目流水线脚本目录。

| 文件 | 作用 | 什么时候运行 |
|---|---|---|
| `00_extract_documents.py` | 从真实 PDF/DOCX/TXT/MD 文件抽取文本 | 放入真实校园资料后运行 |
| `01_prepare_data.py` | 清洗 FAQ 数据并生成文本块 | 改了知识库后运行 |
| `02_build_index.py` | 建立 TF-IDF 索引并记录训练日志 | 改了文本块后运行 |
| `03_evaluate.py` | 跑检索与回答评测，生成图表 | 改了数据或算法后运行 |
| `build_reports.py` | 生成选题报告和中期报告 | 评测结果变化后运行 |

### 9.4 `data/`

数据目录。

| 路径 | 作用 |

## 12. 答辩时可以怎么讲

可以按这个顺序讲：

1. 我们做的是 B07 RAG 校园智能问答助手，目标是解决校园办事、学习生活类问题。
2. 普通大模型容易编造，RAG 的思路是先检索学校资料，再根据资料回答。
3. 我们建立了 60 条校园 FAQ 知识库，并设计了 50 道评测问题。
4. 系统流程包括数据清洗、文本切分、索引构建、检索、回答生成和评测分析。
5. 检索策略方面，我们比较了 TF-IDF、BM25 和 Hybrid 混合检索。
6. 实验结果显示，Hybrid 检索在当前评测集上 Hit@1、Hit@3、MRR 都达到 1.00。
7. 和无检索基线相比，RAG 回答能给出更具体的流程、地点、材料和引用来源。
8. 后续工作是接入更多学校真实 PDF/Word 资料，并把知识库规模和评测问题继续扩大。

如果老师问“为什么不用直接问大模型”，可以回答：

直接问大模型可能不知道本校最新规定，也可能编造流程。RAG 把学校资料作为回答依据，可以提高可追溯性和准确性。

如果老师问“Hybrid 为什么更好”，可以回答：

TF-IDF 更关注文本相似度，BM25 更关注关键词命中。混合检索把两者的分数融合，既能匹配相似问法，也能保留关键词检索的稳定性。

如果老师问“当前项目有什么不足”，可以回答：

当前知识库主要是结构化 FAQ，真实 PDF/Word 资料还需要继续补充；默认回答方式是抽取式回答，后续可以接入通义千问等大模型生成更自然的回答。

## 13. 常见问题

### 13.1 为什么项目不用 API Key 也能运行

因为当前默认是抽取式回答。它先检索知识库，然后把最相关资料组织成答案。  
这样做的好处是课程验收时更稳定，不会因为 API 额度、网络或 Key 配置失败而跑不起来。

### 13.2 为什么要有 `baseline_answer`

这是为了对比实验。  
没有对比，老师很难判断 RAG 是否真的有效。  
有了无检索基线，就可以说明 RAG 的回答更具体、关键词覆盖更高。

### 13.3 为什么要切分文本

因为如果一整篇资料很长，检索时很难精确定位。  
切成文本块后，系统可以只拿最相关的小片段去回答。

### 13.4 为什么要评测 Hit@1、Hit@3、MRR

这些指标用于衡量“检索是否把正确资料排在前面”。

- Hit@1 高：说明第一条结果就经常是正确资料。
- Hit@3 高：说明前三条里基本能找到正确资料。
- MRR 高：说明正确资料通常排得很靠前。

### 13.5 为什么要保留报告 PDF 和 DOCX

DOCX 方便继续修改，PDF 方便直接提交和打印。  
两份报告都由 `scripts/build_reports.py` 根据当前评测结果生成。
